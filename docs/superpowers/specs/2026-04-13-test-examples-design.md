# Test Examples — Design Spec

## Overview

Create 5 new example agents that exercise untested features: postgres sessions, long-term memory, MCP servers, and Python code-first definitions. Each example is a realistic scenario focused on one feature, deployable with `agentstack apply` on Docker using MiniMax via Anthropic-compatible API.

## Examples

### 1. `examples/minimal`

**Scenario:** The simplest possible agent — a chatbot with no tools, no sessions, no services. Just a model and a prompt.

**Purpose:** Tests the minimum viable deployment path. Proves an agent can deploy with nothing but a name, model, and channel.

**Files:**
- `agentstack.yaml`

```yaml
name: minimal-agent
instructions: |
  You are a minimal agent. You have no tools and no memory.
  Just chat and be helpful.
model:
  name: minimax
  provider:
    name: anthropic
    type: anthropic
  model_name: MiniMax-M2.7
  parameters:
    temperature: 0.7
    anthropic_api_url: https://api.minimax.io/anthropic
platform:
  name: docker
  type: docker
  provider:
    name: docker
    type: docker
channels:
  - name: api
    type: api
secrets:
  - name: ANTHROPIC_API_KEY
port: 8090
```

**What it tests:**
- Deploy with no tools, no sessions, no memory, no services
- Bare minimum agent definition
- Verify the generated code handles the absence of all optional features

### 2. `examples/sessions-postgres`

**Scenario:** A conversational assistant with postgres-backed session persistence. Conversations survive container restarts.

**Purpose:** Tests postgres session store provisioning and the new `sessions` field.

**Files:**
- `agentstack.yaml`

```yaml
name: sessions-agent
instructions: |
  You are a helpful assistant with persistent memory of our conversation.
  If the user has told you something before, remember it.
  Refer back to earlier parts of the conversation when relevant.
model:
  name: minimax
  provider:
    name: anthropic
    type: anthropic
  model_name: MiniMax-M2.7
  parameters:
    temperature: 0.7
    anthropic_api_url: https://api.minimax.io/anthropic
platform:
  name: docker
  type: docker
  provider:
    name: docker
    type: docker
sessions:
  type: postgres
  provider:
    name: docker
    type: docker
channels:
  - name: api
    type: api
secrets:
  - name: ANTHROPIC_API_KEY
port: 8091
```

**What it tests:**
- Postgres container provisioning via `sessions` field (not the deprecated `resources`)
- Session persistence across requests (same session_id)
- Session persistence across container restarts (destroy + apply, resume session)

### 3. `examples/memory-agent`

**Scenario:** A personal assistant that remembers facts about you across sessions. Uses both session persistence (conversation history) and long-term memory (facts recalled across different conversations).

**Purpose:** Tests long-term memory with save/forget tools and memory recall at conversation start.

**Files:**
- `agentstack.yaml`

```yaml
name: memory-agent
instructions: |
  You are a personal assistant with long-term memory.
  Remember important facts the user tells you (name, preferences, projects).
  When starting a new conversation, recall what you know about the user.
  Use save_memory when the user shares important facts.
  Use forget_memory when asked to forget something.
model:
  name: minimax
  provider:
    name: anthropic
    type: anthropic
  model_name: MiniMax-M2.7
  parameters:
    temperature: 0.7
    anthropic_api_url: https://api.minimax.io/anthropic
platform:
  name: docker
  type: docker
  provider:
    name: docker
    type: docker
sessions:
  type: postgres
  provider:
    name: docker
    type: docker
memory:
  type: postgres
  provider:
    name: docker
    type: docker
skills:
  - name: assistant
    tools: []
channels:
  - name: api
    type: api
secrets:
  - name: ANTHROPIC_API_KEY
port: 8092
```

**What it tests:**
- Both `sessions` and `memory` fields used together
- Two postgres containers provisioned (one for sessions, one for memory)
- Memory save/forget tools auto-generated by the adapter
- Memory recall at conversation start
- Facts persist across different sessions (new session_id, old facts recalled)

### 4. `examples/mcp-files`

**Scenario:** A file analyst agent that reads files from a directory via the Filesystem MCP server. Can list, read, and search files.

**Purpose:** Tests MCP server integration with stdio transport.

**Files:**
- `agentstack.yaml`
- `sample-docs/readme.txt` — a small sample file for the agent to read
- `sample-docs/notes.txt` — another sample file

```yaml
name: mcp-files-agent
instructions: |
  You are a file analysis agent. You can read files from the /docs directory
  using your MCP tools. When asked about files, use your tools to read and
  analyze them. List files before reading to know what's available.
model:
  name: minimax
  provider:
    name: anthropic
    type: anthropic
  model_name: MiniMax-M2.7
  parameters:
    temperature: 0.3
    anthropic_api_url: https://api.minimax.io/anthropic
platform:
  name: docker
  type: docker
  provider:
    name: docker
    type: docker
mcp_servers:
  - name: filesystem
    transport: stdio
    command: npx -y @modelcontextprotocol/server-filesystem /docs
    install: npm install -g @modelcontextprotocol/server-filesystem
channels:
  - name: api
    type: api
secrets:
  - name: ANTHROPIC_API_KEY
port: 8093
```

**What it tests:**
- MCP server configuration in agent definition
- `install` command gets added to Dockerfile (npm install)
- `command` runs as stdio transport
- MCP tools available to the agent at runtime
- File reading through MCP protocol

**Note:** The `sample-docs/` directory needs to be mounted or copied into the container at `/docs`. The Docker provider already handles workspace mounts — if not, the files can be copied into the build context. The adapter's MCP integration handles tool discovery from the MCP server at startup.

### 5. `examples/code-first`

**Scenario:** Same as `memory-agent` but defined in Python instead of YAML. Demonstrates the programmatic API.

**Purpose:** Tests that Python-defined agents deploy identically to YAML-defined ones.

**Files:**
- `agentstack.py` — Python agent definition (CLI convention filename, auto-discovered)
- `agentstack.yaml` — NOT present (this example uses Python)

```python
# agentstack.py
import agentstack as ast

anthropic = ast.Provider(name="anthropic", type="anthropic")
docker = ast.Provider(name="docker", type="docker")

model = ast.Model(
    name="minimax",
    provider=anthropic,
    model_name="MiniMax-M2.7",
    parameters={
        "temperature": 0.7,
        "anthropic_api_url": "https://api.minimax.io/anthropic",
    },
)

agent = ast.Agent(
    name="code-first-agent",
    instructions=(
        "You are a personal assistant defined in Python code.\n"
        "Remember important facts the user tells you.\n"
        "Use save_memory and forget_memory tools as needed."
    ),
    model=model,
    platform=ast.Platform(name="docker", type="docker", provider=docker),
    sessions=ast.Postgres(provider=docker),
    memory=ast.Postgres(provider=docker),
    skills=[ast.Skill(name="assistant", tools=[])],
    channels=[ast.Channel(name="api", type=ast.ChannelType.API)],
    secrets=[ast.Secret(name="ANTHROPIC_API_KEY")],
    port=8094,
)
```

**What it tests:**
- Python code-first agent definition
- CLI auto-discovers `agentstack.py` (already supported — convention filename)
- All new schema types used programmatically: `ast.Postgres`, `ast.Platform`, `ast.Provider`
- Deploys identically to YAML equivalent

**No CLI changes needed.** The loader already supports `agentstack.py` as a convention filename and handles Python files via `importlib`. It looks for an `agent` variable of type `Agent`.

## Port Assignments

| Example | Port | Agent Name |
|---------|------|------------|
| minimal | 8090 | minimal-agent |
| sessions-postgres | 8091 | sessions-agent |
| memory-agent | 8092 | memory-agent |
| mcp-files | 8093 | mcp-files-agent |
| code-first | 8094 | code-first-agent |

Existing examples use 8080-8083, so these start at 8090 to avoid conflicts.

## MCP File Mounting

The `mcp-files` example needs sample files available inside the container. Two approaches:

1. **Copy into build context** — the adapter already copies all generated files into `.agentstack/<name>/`. We add the `sample-docs/` directory to the build context and a `COPY sample-docs /docs` line to the Dockerfile.
2. **Docker volume mount** — mount a host directory into the container at runtime.

Approach 1 is simpler and self-contained. The Docker provider can handle this by detecting workspace/file dependencies in the agent definition. For this example, we'll copy the sample-docs into the build context manually (the provider already copies tool files this way).

## Testing Each Example

Each example should be testable with:

```bash
cd examples/<name>
export ANTHROPIC_API_KEY=...
agentstack apply
curl http://localhost:<port>/health
curl -X POST http://localhost:<port>/invoke -H "Content-Type: application/json" -d '{"message": "..."}'
agentstack destroy
```

## Out of Scope

- Automated integration test suite (would be nice but not in this spec)
- New service types (Redis, Qdrant examples)
- Bring-your-own connection string example (future)
- OpenAI/other model provider examples
